How Som runs Omni day to day

Hey Tim, this is how Som uses Omni

Omni is a practical operating system, not a demo. Som uses it to turn scattered work into repeatable execution, with Kimi orchestrating, Codex shipping heavy work, and Llama handling low-cost heartbeat checks.

3-agent stack

Kimi (brain), Codex (muscle), Llama (heartbeat + hygiene).

2 operating loops

Real-time session loop + background cron and heartbeat loop.

1 rule

Ship complete outputs with validation and clear status reporting.

At a glance

What this setup is optimized for

Speed with guardrails

Fast execution, but with pre-send QA gates and explicit completion receipts.

Parallel heavy lifting

Codex handles coding, file processing, and synthesis while Kimi keeps context and priorities straight.

Quiet autonomy

Heartbeat and cron checks catch urgent items without spamming Som all day.

Feature demos

Interactive demos of Som's routine

Morning triage

Heartbeat checks pull urgent email/calendar signals, then Kimi prioritizes only what needs attention now.

  • Read alerts and detect urgency windows
  • Draft concise action plan for Som
  • Spawn Codex only for heavier tasks

Workday execution

Codex runs production-grade implementation and validation while Kimi manages external communication safely.

  • Multi-file coding and documentation updates
  • Lint/build/test before reporting done
  • Status reports with blockers and next action

Evening cleanup

State gets persisted: daily memory logs are condensed into long-term memory when it matters.

  • Capture durable decisions and preferences
  • Close loops with completion receipts
  • Prepare next-day heartbeat checklist

Any coding task, multi-file analysis, or 10+ turn effort gets delegated. Kimi stays focused on orchestration, while Codex executes deeply and reports back with validation.

Read/search/internal edits are free. Anything that sends data outside the machine is gated and checked, especially in group contexts.

Pre-send QA validates budget caps, mandatory sections, banned phrasing, and file naming before any user-facing send action.

Roles

Who does what in the stack

Kimi

Maintains context, routes work, and guards external actions.

  • Loads SOUL/USER/memory state each session
  • Coordinates multiple agents
  • Keeps response quality and tone aligned

Codex

Builds and validates concrete outputs fast.

  • Coding, debugging, and project setup
  • Parallel file analysis and refactors
  • Reports validation and blockers clearly

Llama

Runs low-cost routine checks and background hygiene.

  • Heartbeat polls and reminders
  • Simple recurring checks
  • Keeps main session signal focused

Tools and automation

Cron, heartbeat, and sub-agent control plane

Cron jobs

Use cron for precise time-based tasks: reminders, exact schedules, and isolated one-shot tasks.

  • Exact schedules: weekly 09:00 reports
  • One-shot reminders: follow up in 20 minutes
  • Isolated jobs with separate model settings

Heartbeat checks

Use heartbeat for grouped periodic checks where slight timing drift is acceptable.

  • Batch email + calendar + mentions in one pass
  • Stay quiet overnight unless urgent
  • Ping only when there is fresh value

Sub-agents

Som routinely spawns focused agents for coding, synthesis, and data-heavy workflows.

  • Always spawn for coding tasks
  • Auto-spawn for multi-file or long-horizon tasks
  • Return completion reports with validation every time
